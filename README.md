SmartClean API

ALX Backend Engineering Capstone Project

A backend REST API built to manage the day-to-day operations of a cleaning business.

📌 Why I Built This
I noticed that most small cleaning businesses are held together by WhatsApp group chats, sticky notes, and shared spreadsheets. Jobs get missed, clients get frustrated, and staff spend more time coordinating than actually cleaning.
SmartClean is my attempt to fix that. It gives a cleaning business a proper system, where clients can request jobs, admins can manage everything, and cleaners know exactly what they're supposed to do and when.
This was built as my capstone project for the ALX Backend EngineeringProgram.

 How Users Work
When you register, you pick who you are :a client, a cleaner, or an admin. That choice follows you everywhere in the system and determines what you can see and do.
| Role    | What they can do                                              |
|---------|---------------------------------------------------------------|
| Admin   | Everything — manage users, assign cleaners, generate quotes   |
| Client  | Request jobs, pick services, view their own jobs              |
| Cleaner | See jobs assigned to them and mark them as completed          |

I used Django Groups to enforce this. It's not just a label — every endpoint checks your group before letting you through.

 What the System Does

Register and login with JWT tokens
Three distinct user roles enforced on every endpoint
Clients and cleaners each have their own profile separate from their login account
A service catalogue that admins manage — clients browse and pick what they need
Jobs go through a proper lifecycle — you can't skip steps or go backwards
Quotes are generated by the admin based on what services were selected
Clients approve the quote before any work is scheduled
Prices are locked at quote time — if a service price changes later, existing quotes are unaffected
Cleaners, clients and services are protected from deletion if jobs are attached to them


 How a Job Moves Through the System
This was one of the more interesting parts to build. A job can't just jump to completed — it has to pass through every stage in order.
draft → quoted → scheduled → in_progress → completed
  ↓        ↓         ↓            ↓
cancelled cancelled cancelled   (too late to cancel)

| Status      | Who triggers it                        |
|-------------|----------------------------------------|
| draft       | Client creates the job                 |
| quoted      | Admin generates a quote                |
| scheduled   | Client accepts the quote               |
| in_progress | Admin starts the job                   |
| completed   | The assigned cleaner marks it done     |
| cancelled   | Admin or the client who owns the job   |

I did this using an ALLOWED_TRANSITIONS dictionary on the Job model. 

 How Pricing Works
Every service in the catalogue has a base price. When a client adds services to their job, the system takes a snapshot of the current price at that moment and stores it on the job. This  means that if an admin later changes the price of a service, it won't affect quotes that have already been generated.
The admin generates a quote by summing up all the service line items: quantity × unit_price. The client sees the full breakdown before they agree to anything.

You Can't Delete Things That Are Still in Use
This tripped me up early on ;Django would throwa database error if you try to delete something that other records depend on. So, I added  checks:

Cleaner cannot be deleted if job exists
If a Client has jobs → it cannot be deleted just like that
A service on a job cannot be deleted, admin should deactivate it instead
Job is not in draft → deletion blocked, cancel it instead
